# Dashboard concept

`assess` produces an `assessment-graph.json`. The dashboard is the first user-facing view after a run: it should make the generated graph understandable without implying that every part of the repository is clean.

## Product principle

The dashboard is not an architecture tour. It is a post-assessment cockpit:

```text
run assess -> validate assessment-graph.json -> open dashboard -> inspect trust, coverage, gaps, findings
```

The first screen must answer four questions immediately:

1. What repository or module was assessed?
2. How much was assessed, and what was not assessed?
3. How many findings exist, and how severe are they?
4. Which graph view should the user inspect next?

## Loading a generated graph

For the bundled sample graph:

```bash
pnpm dev:dashboard
```

For a graph generated by an assessment run:

```bash
node packages/core/runtime/assess-run.mjs --repo packages/core/src --out .assessment
ASSESS_GRAPH="$(pwd)/.assessment/assessment-graph.json" pnpm dev:dashboard
```

When `ASSESS_GRAPH` is set, the Vite dev server exposes that file as `/__assessment_graph__`. The browser loads the exact generated artifact instead of the bundled sample graph.

The dashboard also accepts a query override:

```text
http://localhost:5180/?graph=/assessment-graph.json
```

Use the query parameter only for URLs the browser can fetch. Use `ASSESS_GRAPH` for local files outside the dashboard public directory.

## Information architecture

### 1. Run summary

Top-level dashboard band. It summarizes the loaded graph:

- project name;
- assessment headline;
- commit binding;
- generation time;
- headline trust;
- assessed percentage;
- coverage, finding, node, and edge counts;
- severity distribution.

This is the user's immediate orientation layer after a run.

### 2. Coverage banner

Second layer. It makes trust boundaries explicit:

- assessed / partial / not assessed / coverage-insufficient counts;
- headline trust;
- explicit coverage gaps.

If coverage is partial, the banner must make that visible before the user inspects findings.

### 3. Graph workspace

The body has two columns:

- left sidebar: filters + selected node/finding information;
- main canvas: active graph/report view.

Supported views:

| View | Purpose | User question |
|---|---|---|
| Coverage map | Show every area and node by coverage status | What was actually assessed? |
| Gap map | Show intent/code/baseline relationships and gap edges | Where does code fail to satisfy intent? |
| Findings | Show ranked evidence-bound verdicts | What should I fix or investigate first? |

## Interaction model

- Click a node to inspect evidence, readiness, ownership, coverage status, and linked findings.
- Click a gap edge to select the linked finding when present.
- Filter by gap direction, minimum severity, satisfied edges, and text search.
- Preserve the same graph artifact across all views; switching tabs changes projection, not data.

## Trust rules

The dashboard must never hide uncertainty.

- `not_assessed` and `coverage_insufficient` must remain visually loud.
- A clean findings list is not equivalent to a clean repository when coverage is partial.
- The graph version and binding should remain visible enough for audit context.
- The dashboard should render only `kind: assessment` graphs.
